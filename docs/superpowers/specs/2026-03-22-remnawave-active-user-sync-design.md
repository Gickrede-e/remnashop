# Remnawave Active User Sync Design

## Summary

Add a safe Remnawave sync flow for site-managed users.

Two behaviors are in scope:

1. Any new Remnawave user created by this site must use a `gs_` username prefix derived from the local-part of the email address.
2. Admins must be able to run a bulk sync from the admin users screen so local users with an `ACTIVE` subscription are created or attached in Remnawave when they are missing there.

The design must not touch `app/api/*` business boundaries outside the new admin sync route, must not rename or mutate existing non-`gs_` Remnawave users, and must not interfere with users that belong to other projects.

## Goals

- Make site-created Remnawave identities predictable through a `gs_*` naming convention.
- Add an admin-triggered sync action for active subscriptions missing in Remnawave.
- Reuse the existing subscription provisioning flow in [lib/services/subscriptions.ts](/home/gickrede/remnashop/.worktrees/feat/remnashop-web/lib/services/subscriptions.ts).
- Reuse the existing Remnawave client in [lib/services/remnawave.ts](/home/gickrede/remnashop/.worktrees/feat/remnashop-web/lib/services/remnawave.ts).
- Keep sync safe in a shared Remnawave instance that may contain users from other projects.

## Non-Goals

- No automatic mass rename of existing Remnawave users.
- No synchronization for `DISABLED`, `EXPIRED`, `PENDING`, or `CANCELED` subscriptions.
- No changes to payment provider flows, API route contracts outside the new admin route, or public site routes.
- No background cron or queue-based sync in this phase.

## Current Context

Existing Remnawave provisioning already happens from [lib/services/subscriptions.ts](/home/gickrede/remnashop/.worktrees/feat/remnashop-web/lib/services/subscriptions.ts) through `ensureRemnawaveIdentity`, `createRemnawaveUser`, and `updateRemnawaveUser`.

Current issues:

- Remnawave usernames are generated by `buildRemnawaveUsername(email, userId)` and do not follow the requested `gs_*` convention.
- Admin UI only supports per-user sync through [app/api/admin/users/[id]/sync/route.ts](/home/gickrede/remnashop/.worktrees/feat/remnashop-web/app/api/admin/users/[id]/sync/route.ts) and [components/admin/user-actions.tsx](/home/gickrede/remnashop/.worktrees/feat/remnashop-web/components/admin/user-actions.tsx).
- There is no bulk repair action for active local subscriptions missing in Remnawave.

Relevant Remnawave API capabilities from [api-1.json](/home/gickrede/remnashop/.worktrees/feat/remnashop-web/api-1.json):

- `POST /api/users` creates a user.
- `PATCH /api/users` updates a user by `uuid` or `username`.
- `GET /api/users/by-username/{username}` fetches one user.
- `GET /api/users/by-email/{email}` returns a list of users.
- Username must match `^[a-zA-Z0-9_-]+$` and be `3-36` characters long.

## Identity Model

### Site-managed username rule

All new Remnawave users created on behalf of a site user must use a `gs_` username derived from the email local-part.

Base algorithm:

1. Take the substring before `@`.
2. Convert to lowercase.
3. Replace characters outside `[a-z0-9_-]` with `_`.
4. Collapse repeated underscores.
5. Trim leading and trailing underscores.
6. Prefix with `gs_`.

If the result is empty, too short, too long, or collides with an existing Remnawave username, generate a safe fallback:

- `gs_{normalizedLocalPart}_{shortId}`

`shortId` is a short deterministic suffix derived from the local user id to reduce collisions.

The generated username must always respect Remnawave constraints:

- only letters, digits, `_`, `-`
- minimum length `3`
- maximum length `36`

### Existing linked users

If a local user already has a linked `remnawaveUuid`, that linkage remains authoritative.

If the linked remote user has a non-`gs_` username, it is not renamed in this phase.

### Existing remote users found by lookup

When searching for a matching remote user by email, only records with usernames starting with `gs_` are eligible for attachment to a local site user.

Non-`gs_` usernames are treated as belonging to other projects and must be skipped.

## Provisioning Flow Changes

### `ensureRemnawaveIdentity`

`ensureRemnawaveIdentity` in [lib/services/subscriptions.ts](/home/gickrede/remnashop/.worktrees/feat/remnashop-web/lib/services/subscriptions.ts) becomes the single safe entrypoint for linking or creating a Remnawave identity.

Expected behavior:

1. If the local user already has `remnawaveUuid`, return it immediately unless the sync flow explicitly detects that the remote record no longer exists.
2. Compute the preferred `gs_*` username from the email and local id.
3. Try `GET /api/users/by-username/{username}` with the preferred username.
4. If found, attach that remote user to the local user and persist `remnawaveUuid`, `remnawaveUsername`, and `remnawaveShortUuid`.
5. If not found, try `GET /api/users/by-email/{email}`.
6. From the email results, only consider users whose `username` starts with `gs_`.
7. If exactly one eligible `gs_` user exists, attach it.
8. If multiple eligible `gs_` users exist, mark the case as skipped because the match is ambiguous.
9. If no eligible user exists, create a new user in Remnawave with the preferred username or fallback username.

This keeps a single attach/create rule shared by:

- payment activation
- admin manual grant
- per-user sync
- bulk active-user sync

### Lost-link recovery

If a local user has `remnawaveUuid`, but fetching `GET /api/users/{uuid}` fails with a not-found result, the link is treated as stale.

In that case, the code should:

1. Clear the assumption that the current uuid is valid for this sync run.
2. Re-run the safe attach/create flow described above.
3. Persist the recovered or recreated identity back to the local user.

## Bulk Sync Design

### Candidate set

The new bulk sync action processes only local users whose subscription status is `ACTIVE`.

This includes users who:

- have no `remnawaveUuid`
- have a stale `remnawaveUuid` whose remote record no longer exists
- are already linked and need a snapshot refresh

Users without an `ACTIVE` subscription are excluded entirely.

### Service contract

Add a new bulk sync method in [lib/services/subscriptions.ts](/home/gickrede/remnashop/.worktrees/feat/remnashop-web/lib/services/subscriptions.ts), for example `syncActiveSubscriptionsToRemnawave`.

It returns a summary object:

- `totalCandidates`
- `created`
- `attached`
- `alreadyLinked`
- `skipped`
- `failed`
- `items`

`items` contains per-user results for debugging and admin feedback. Each item should include at least:

- `userId`
- `email`
- `outcome`
- `message`

### Outcome definitions

- `created`: user was missing and a new `gs_*` Remnawave account was created
- `attached`: user was missing locally but a safe existing `gs_*` remote account was found and linked
- `alreadyLinked`: user already had a valid local-to-remote mapping and sync only refreshed state
- `skipped`: user was intentionally not attached because the remote matches were unsafe or ambiguous
- `failed`: a local or remote error prevented sync for that user

### Execution model

Bulk sync must not fail all users because of one bad record.

Implementation rules:

- process users independently
- limit concurrency to a small batch size
- aggregate counts at the end
- continue after per-user failures

## Safety Rules

These rules are mandatory to avoid cross-project contamination in Remnawave:

- never rename existing non-`gs_` remote users
- never auto-attach a non-`gs_` remote user found by email
- if email lookup returns multiple `gs_` matches, skip instead of guessing
- if email lookup returns only non-`gs_` matches, skip
- if username lookup finds a `gs_*` user, it is safe to attach

## Error Handling

### Per-user sync

Per-user sync remains best-effort and should continue to refresh subscription state from Remnawave where possible.

If a user is missing in Remnawave but has an active local subscription, the flow should recover through the same attach/create logic.

### Bulk sync

Per-user errors must be isolated.

Each failed user should contribute:

- a `failed` outcome
- a human-readable error message in the result item

The overall bulk call should still succeed if some users were processed successfully.

## Admin Logging

Keep the existing per-user log behavior for `SYNC_USER`.

For bulk sync, write one aggregate admin log event, for example:

- `SYNC_ACTIVE_USERS`

Store counts and identifiers for skipped or failed records in the log `details`.

## Testing

### Automated tests

Add tests for:

1. username generation
2. attach by expected `gs_*` username
3. attach by email only when the remote record is `gs_*`
4. skip when email lookup returns only non-`gs_` users
5. skip when email lookup returns multiple `gs_` matches
6. create when nothing is found
7. recreate or reattach when stored `remnawaveUuid` no longer resolves
8. bulk route returns summary counts

### Manual verification

Verify these scenarios manually:

1. A newly provisioned site user gets a `gs_*` username in Remnawave.
2. Bulk sync from admin users creates a missing remote user for a local `ACTIVE` subscription.
3. A user from another project with a non-`gs_` username is not attached.
4. The admin summary matches actual changes in Remnawave.
5. Per-user sync still works after the shared attach/create logic is updated.
